STABILISATION AUDIT: API ENGINE + REDIS + WEBUYANYHOUSE LEAD COUNTS + ANALYTICS + CAMPAIGN/AGENT PREP

We have added Redis and the API Engine. Webuyanyhouse lead numbers and analytics are now inconsistent.

Do not rebuild pages.

Do not create new pages.

Do not change business logic yet.

Do not change sentiment/qualification rules yet.

Do not start implementing campaigns or agent APIs yet.

Goal:

Find exactly why lead, call, data and analytics numbers differ between:

* Source API
* API Engine
* Redis cache
* Supabase/local cache if used
* UI pages
* Dashboard analytics

Also prepare a clean audit foundation for adding Campaign APIs and Agent APIs later today.

---

For every Webuyanyhouse API endpoint currently used, log safely:

* endpoint called

3. REDIS CACHE AUDIT

---

Check Redis behaviour for every module:

* Dashboard analytics
* Leads
* People
* Calls
* Qualified Leads
* Data page and sub-tabs
* Campaigns prep
* Agents prep

Check:

* cache key format
* workspace_id included
* module included
* endpointMappingId included
* page included
* pageSize included
* filters included
* search included
* sort included
* sentiment/status filters included
* date range included for analytics
* campaign/user filters included if present
* TTL value
* stale cache usage
* cache invalidation after sync
* whether page 1 cache is reused for other pages
* whether analytics cache is reused across different date ranges
* whether WBAH cache is shared with another workspace by mistake

Required cache key shape example:

webee:data:v1:{workspaceId}:{module}:{endpointMappingId}:{page}:{pageSize}:{filtersHash}:{sortHash}:{dateRangeHash}

Analytics cache key example:

webee:analytics:v1:{workspaceId}:{metric}:{endpointMappingId}:{startDate}:{endDate}:{filtersHash}

Campaign cache key example:

webee:campaigns:v1:{workspaceId}:{endpointMappingId}:{page}:{pageSize}:{filtersHash}

Agent cache key example:

webee:agents:v1:{workspaceId}:{endpointMappingId}:{filtersHash}

7. SPECIFIC WBAH RULES

---

For Webuyanyhouse:

* Leads page should show only positive and negative sentiment if that is the current configured rule
* Qualified Leads should show positive sentiment only
* Neutral must not count as qualified
* Partly Qualified should be neutral only
* Unknown should not be counted as qualified

Verify these rules are applied in one place only, preferably config/data service, not duplicated in multiple components.

Do not change the rules during this audit unless a bug is directly identified and the fix is safe.

---

8. REDIS SAFETY FIXES IF NEEDED

---

If Redis is causing mismatch:

* add a development-only bypass flag
* add cache versioning
* add manual cache clear for workspace/module
* ensure cache keys include workspaceId, module, endpointMappingId, page, pageSize, filters and date range
* do not cache incomplete first-page results as full totals
* do not cache error/empty responses as valid data unless explicitly marked
* do not share analytics cache across date ranges
* do not share WBAH cache with another workspace
