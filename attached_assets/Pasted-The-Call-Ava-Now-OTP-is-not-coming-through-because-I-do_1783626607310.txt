## 4. Email OTP fallback

If SMS is not configured but Resend is configured, send the OTP to the user’s email address.

Email subject:

`Your Ava verification code`

Email body:

“Your verification code for speaking with Ava is: {{OTP}}

This code expires in 10 minutes.

If you did not request this, you can ignore this email.”

Important:

* Generate OTP server-side.
* Store only hashed OTP.
* Do not store raw OTP.
* Do not return OTP in API response.
* Do not log raw OTP in production.
* The OTP should expire after 10 minutes.
* OTP should be one-time use only.
* Limit attempts to 5.

Frontend message if email OTP is used:

“We’ve sent your verification code to your email.”

If SMS is configured later and email fallback is used because SMS failed:

“We could not send SMS, so we sent the code to your email.”

`POST /api/public/ava-call/request-otp`

Expected payload:

```json
{
  "name": "Full name",
  "email": "email@example.com",
  "phone": "+447000000000",
  "businessWebsite": "https://example.com",
  "consent": true
}
```

Backend should:

1. Validate name, email, phone, and consent.
2. Normalize phone number.
3. Rate limit by IP, email, and phone.
4. Detect available OTP provider.
5. Generate OTP.
6. Store hashed OTP.
7. Send OTP using available provider.
8. Return which channel was used, without exposing secrets.

Example response for email OTP:

```json

  "success": true,
  "channel": "email",
  "message": "Verification code sent to your email."
```

Example response for SMS OTP:

```json
  "success": true,
  "channel": "sms",
  "message": "Verification code sent to your phone."
}
```

Example response if no provider is configured:

```json

  "success": false,
  "message": "Verification is not configured yet. Please book a demo instead."
```

5. Enforce attempt limits.

7. Trigger Ava call server-side using Retell.
