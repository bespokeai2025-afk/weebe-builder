Do NOT check Retell.
Do NOT add Retell fallback.

API base URL:
https://uat-api.webespokeai.com

Goal:
Find out why WEBEE is only showing around 50 leads/calls/CRM records instead of the full dataset from the API.

Do not rebuild pages.
Do not create new pages.
Do not change unrelated modules.

Perform a full end-to-end audit of the API data flow only.

Check these areas:

For each endpoint, log safely:

- endpoint

- whether response includes total count
- whether response includes pagination metadata
- whether response includes page
- whether response includes limit

4. Pagination audit

Determine if the API returns only the first page.

If response includes pagination metadata, implement fetching all pages server-side.

Examples of possible pagination shapes:

{
  data: [...],
  total: 10000,

  totalPages: 200
}

  }
}

or:

{
  data: [...],
  nextCursor: "..."
}

or:

{
  data: [...],
  nextPage: 2

Support the actual response shape returned by this API.

5. Count mismatch audit

Compare:

GET /call-output-data/get-call-count

against:

GET /call-output-data/get-all-calldata

If get-call-count says there are thousands but get-all-calldata returns 50, then pagination is confirmed.

Report this clearly.

6. CRM count audit

Check whether:

GET /crm-data/get-crm-data

returns pagination metadata.

If it only returns 50, identify whether that is:

- API default
- frontend limit
- server adapter limit
- table display limit

7. Dashboard count audit

Check whether dashboard cards are using count endpoints or only counting the 50 returned rows.

Dashboard cards must not calculate totals from only the first page.

If totals exist from dashboard endpoints, use those directly.

8. React Query audit

Check all React Query hooks for People, Calls, CRM and Dashboard.

Verify:

- queryKey includes workspaceId and pagination/date filters
- queryFn is calling the correct server route
- enabled is not false accidentally
- only first page is not being treated as full dataset
- pagination metadata is preserved
- stale cached 50-row response is not being reused globally

9. Server adapter audit

Check the Webuyanyhouse API client/server adapter.

Identify whether it:

- normalizes only first 50 rows
- drops pagination metadata
- slices data
- deduplicates incorrectly
- maps only first page
- returns data.data instead of full response with meta

10. Table display audit

Check People, Calls and CRM tables.

Determine:

- how many records the API returns to the component
- how many records the component receives
- how many records the table displays
- whether pagination is client-side or server-side
- whether table default page size is 50

If the table page size is 50 but total pagination works, that is okay.

If the table only has 50 records total, fix the data fetch.

11. Implement safe fix

If API supports pagination:
- implement server-side pagination correctly
- expose total count to frontend
- make table request page changes properly

If API supports larger limit:
- use a safe page size like 100 or 250
- do not request unlimited records into browser

If API requires all-page sync:
- create server-side function fetchAllPages()
- loop until all pages/cursors are fetched
- cache/store results if needed
- frontend should still paginate display
