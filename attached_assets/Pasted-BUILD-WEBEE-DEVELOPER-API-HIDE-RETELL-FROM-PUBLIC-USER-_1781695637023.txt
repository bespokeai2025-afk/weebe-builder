BUILD WEBEE DEVELOPER API + HIDE RETELL FROM PUBLIC USER VIEW

IMPORTANT

Do not break Retell functionality.

Do not remove Retell internally.

Do not break existing agents, calls, webhooks, provider framework or billing.

Retell must remain available internally as infrastructure, but normal customers should not see “Retell” as the provider brand.

OBJECTIVE

1. Add customer-facing API endpoints so external systems can connect to WEBEE.
2. Build a developer/API access layer.
3. Hide Retell from public/customer UI and replace it with WEBEE-branded voice infrastructure.

====================================================

CUSTOMER-FACING API ENDPOINTS

Create public API endpoints:

POST /api/v1/leads

Create lead/contact in WEBEE.

POST /api/v1/calls/trigger

Trigger an AI call using a selected WEBEE agent.

POST /api/v1/campaigns/trigger

Trigger a campaign or enrol a lead into a campaign.

POST /api/v1/agents/{agentId}/call

Trigger outbound call for a specific agent.

GET /api/v1/agents

List workspace agents.

GET /api/v1/calls

List call logs.

GET /api/v1/leads

List leads.

POST /api/v1/knowledge/upload

Upload knowledge document.

POST /api/v1/webhooks/test

Send test webhook.

====================================================

AUTHENTICATION

All /api/v1/* endpoints require:

Authorization: Bearer <WEBEE_API_KEY>

Validate:

token exists
not revoked
not expired
workspace scoped
permissions allow action
