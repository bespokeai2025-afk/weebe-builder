Please extend the Account Settings, Campaign Reports, Notifications, and Team Access system.

I want each workspace to have account-level settings where authorised users can choose who gets notified when important campaign events happen, and I want workspace admins to be able to give staff different levels of access to the WEBEE app.

This is especially important for WBAH, where different staff members may need different access levels to leads, calls, campaigns, reports, CRM data, and settings.

Everything must be workspace-scoped and must never affect another workspace.

---

# 1. Account Settings: Notification Preferences

Add a section inside Account Settings called:

`Notifications`

or

`Campaign Notifications`

Workspace admins should be able to configure email notifications for campaign events.

Notification events should include:

* campaign launched
* campaign activated
* campaign paused
* campaign completed
* campaign failed
* campaign blocked
* campaign has no eligible leads
* campaign hits daily call cap
* campaign hits safety cap
* campaign has provider/API error
* campaign has workflow error
* campaign has KPI report available
* campaign has high negative sentiment
* campaign has high positive performance
* campaign generated qualified leads
* campaign booked appointments
* campaign created follow-up tasks
* campaign needs admin attention

---

# 2. Notification recipients

Allow the workspace admin to choose who receives notifications.

Options:

* workspace owner
* workspace admins
* selected users
* selected roles
* custom email addresses
* campaign owner
* assigned agent manager
* AccountsMind users
* HiveMind/SystemMind users where relevant

Each event type should allow different recipients.

Example:

Campaign failed:

* owner
* admins
* campaign manager

Campaign KPI report:

* owner
* admins
* AccountsMind users

Campaign launched:

* owner
* campaign manager

# 5. Email sending

Use the existing email provider already configured in WEBEE, for example Resend if present.

Do not add a second email system if one already exists.

Email notifications should include:

* workspace name
* campaign name
* campaign status
* event type
* summary
* KPI highlights where relevant
* failure reason where relevant
* recommended action where relevant
* link to Campaign Report
* link to Campaigns page

If email sending fails:

* mark notification as failed
* do not break campaign execution
* do not retry infinitely

Campaign execution must never fail because notification email failed.

# 9. Role levels

Support role-based access.

Base roles:

### Owner

Full workspace access.

Can:

* manage billing
* manage settings
* manage users
* approve high-risk actions
* deploy/go live
* manage provider keys
* manage campaigns
* view all reports
* manage SystemMind/HiveMind/GrowthMind/AccountsMind

### Admin

Full operational access, but billing/provider secrets can be restricted if needed.

Can:

* manage campaigns
* approve workflows
* deploy agents
* manage staff below admin
* view reports
* edit CRM settings

### Manager

Operational management access.

Can:

* view leads/calls/qualified
* manage campaigns if permitted
* view campaign reports
* create draft workflows
* request approval
* use SystemMind in draft mode
* cannot approve high-risk actions unless explicitly allowed

### Agent Builder

Can:

* build/edit draft agents
* create workflow drafts
* run test calls
* request approval
* cannot go live unless permitted
* cannot manage billing/provider keys

### Campaign Manager

Can:

* create campaign drafts
* run campaign dry-runs
* view campaign reports
* request campaign activation
* cannot override safety blocks unless admin allows

### Reports Only

Can:

* view reports
* view campaign KPIs
* view AccountsMind summaries if permitted
* cannot edit data or campaigns

### Viewer

Read-only access.

### No Access / Suspended

Cannot access workspace.

# 18. Safety

Do not break current login or workspace membership logic.

Do not break existing owner/admin/member/viewer roles.

Map existing roles into the new permission model carefully.

If a workspace has no custom permissions, use safe defaults.

If permission system fails, do not accidentally grant more access. Fail closed.

Critical:

* never allow a lower role to grant themselves higher permissions
* never allow cross-workspace invitations
* never expose provider keys to unauthorised users
* never expose billing to unauthorised users
* never allow WBAH staff to access other workspaces
* never allow other workspace users to access WBAH
