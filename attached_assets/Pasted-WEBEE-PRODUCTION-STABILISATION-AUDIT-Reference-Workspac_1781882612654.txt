Issues Identified
Issue L-1 (Critical): The /leads page has no source filter. It shows ALL rows in the WBAH workspace's leads table, including CRM contacts that may have been written there. Seller count, CRM contact count, and any other imported rows are all merged into one list with no way to distinguish them.

Issue L-2 (Critical): "Qualified" count is defined differently across pages:

Dashboard KPI: sentiment = 'positive' only
Qualified page: sentiment = 'positive' hardcoded (consistent with dashboard)
Contacts page (listWbahLeads): excludes sentiment = 'negative' — includes positive + neutral + empty → higher count
This means three different pages could show three different "qualified" numbers for the same workspace.
Issue L-3 (High): /leads page default date filter is Last 30 days. Older WBAH leads (before ~20 May 2026) are invisible unless the user manually expands the date range. Since the API has 1,568 leads and sync started recently, many leads will fall outside the default window.

Issue L-4 (High): No soft-delete (is_deleted) check on leads, wbah_calls, or wbah_categorized_leads. If a record is logically deleted at the source, it still appears in all queries on sync.

Issue L-5 (Medium): /pipeline page fetches all leads with no date filter and no pagination. As the WBAH workspace grows, this becomes a full-table scan and will OOM or timeout.

Issue L-6 (Medium): listWbahLeadsForPeople filters CRM contacts in application memory (.filter(r => r.meta?.wbah_source !== "crm")) after fetching all rows from DB. At 1,568 rows this is acceptable; at 10,000+ rows this wastes memory and slows the response.

SECTION 3 — WEBUYANYHOUSE LEAD INVESTIGATION
Why Leads Are Missing / Count Inconsistency
Finding 3-A — Sync Not Complete at Dashboard Load Time:
Live logs show the sync running in incremental waves:

[wbah-leads-sync] fetchAllLeadRecords done: 300 unique records   (wave 1)
[wbah-leads-sync] fetchAllLeadRecords done: 1300 unique records  (wave 2)
[wbah-leads-sync] fetchAllLeadRecords done: 1550 unique records  (wave 3)

The WeeBespoke API reports totalItems=1568. The sync batches API pages 5-at-a-time with 200ms delays. Each full sync cycle takes multiple minutes. If a user opens the dashboard mid-sync, they see a partial count (e.g., 300 instead of 1,568).

Finding 3-B — WeeBespoke API totalPages Bug:
The API's pagination.totalPages field is wrong (returns 13 when actual is 203+ for pageSize=10). The sync correctly computes Math.ceil(totalItems / pageSize) as a workaround. However if totalItems itself is ever wrong (e.g., delayed update at source), the sync will miss records.

Finding 3-C — Three Different Lead Counts Displayed Simultaneously:

Surface	Count Source	Likely Count
Dashboard "Total Leads" KPI	All rows in leads table, workspace_id = WBAH, excluding not_interested	~1,550
/leads page	Same, but filtered to Last 30d by default	Much lower
People tab "Leads"	source_detail = webespoke_enterprise + not CRM in meta	~1,500
Dashboard "Qualified" KPI	sentiment = positive + not not_interested	?
Contacts page	sentiment != negative	Higher than dashboard qualified
Finding 3-D — CRM Contacts Stored in leads Table:
The sync upserts buyer CRM contacts into data_records. However, earlier code paths or full-resync operations may have written some buyer contacts into the leads table with meta.wbah_source = "crm". These would inflate the /leads page count while being filtered out of the People tab. The exact count of such misplaced rows is unknown without a live DB query.

Finding 3-E — Status Classification Keyword Fallback:
classifyStatus() uses keyword searching ("disqualif", "tried contact", "qualified") as a last resort. If a lead's WeeBespoke status string doesn't match any known pattern, the lead gets no status classification — it may be classified as need_to_call by default, effectively hiding it from filtered views.

Finding 3-F — No created_at Set From Source:
Leads upserted from WeeBespoke don't carry the original WeeBespoke creation date into leads.created_at. Supabase sets created_at to the time of the upsert (first sync time). This means:

The "Last 30 days" filter on the Leads page is actually "imported in the last 30 days," not "created at source in the last 30 days."
All 1,568 leads will have created_at set to approximately when the sync first ran — so they all appear within the same day. This makes date filtering misleading.

Finding 4-A (Critical): The /calls page uses listCalls which always queries the calls table. There is no WBAH branch. The wbah_calls table is never touched. WBAH users who navigate to /calls see zero records despite having 10,966 calls in the database.

Finding 4-B (High): wbah-calls-sync plugin documentation says "runs every 4 hours" but the code sets INTERVAL_MS = 5 * 60 * 1000 (5 minutes). This is a documentation/intent mismatch. Running every 5 minutes on 10,966+ calls (1,096+ API pages in batches of 20) means the sync likely doesn't finish before the next tick starts, potentially creating concurrent sync runs.

Finding 4-C (High): getRetellAnalytics hardcodes in_voicemail = false for all WBAH calls. If WeeBespoke marks any calls as voicemail at source, this information is lost in the analytics layer.

Finding 4-D (High): wbah_calls has only a SELECT RLS policy — no INSERT, UPDATE, or DELETE policies. The sync uses supabaseAdmin (service role) which bypasses RLS, so upserts work. But any future code using the standard client to write to wbah_calls will silently fail.

Finding 4-E (Medium): The wbah-calls-sync derives id from call_id ?? _id ?? id. If any of these fields are missing from the API response, the upsert conflict key is undefined, creating a new row each tick (duplicate explosion risk).

Finding 4-F (Medium): People > Calls tab renders r.wbah_name and r.wbah_contact from wbah_calls, but the actual columns written by the sync are customer_name and phone. This means all Name and Contact cells in the Calls tab display as "—" despite the data existing in the DB.

Finding 6-A (High): getRetellAnalytics (analytics page) has a 15-minute TTL and is never explicitly invalidated. If calls finish during that window, the analytics page shows stale sentiment/call counts for up to 15 minutes. For WBAH, this means new calls synced into wbah_calls won't appear in analytics for up to 15 minutes.

Finding 6-B (High): invalidateDashboardCache deletes dashboard, hivemind, and growthmind keys. It does not invalidate calls cache keys. A new call finishing and being written to wbah_calls does not flush any cache — the calls list stays stale for its TTL.

Finding 6-C (Medium): Module-level variable _wbahReloginAt throttles WeeBespoke token refresh to once per 30 minutes. In a serverless/frequent-restart environment (dev with HMR, or prod deployments), this module state resets on every restart — losing the throttle state and potentially causing burst re-login requests.

Finding 6-D (Low): Redis graceful degradation (UPSTASH_REDIS_REST_URL missing → no cache) means if Redis goes down in production, all requests hit Supabase directly. This is correct behaviour but the sudden load spike is not rate-limited.

Finding 7-A (Critical): wbah_categorized_leads table does not exist. The wbah-category-sync plugin runs every 30 minutes and fails every time with: Could not find the table 'public.wbah_categorized_leads' in the schema cache. Migration WBAH_CATEGORY_LEADS_MIGRATION.sql has never been applied.

Finding 7-B (High): wbah_calls has SELECT-only RLS. No INSERT/UPDATE/DELETE policies exist. The sync plugin uses supabaseAdmin (service role, bypasses RLS) so it works today. If any future code writes to wbah_calls using the user-scoped client, it will fail silently.

Finding 7-C (High): calls.lead_id uses ON DELETE SET NULL. When a lead is deleted, associated calls lose their lead linkage. These now-orphaned calls still count in analytics queries (workspace_id is preserved). This can cause analytics to show calls for leads that no longer exist.

Finding 7-D (Medium): retell_webhook_events.workspace_id is nullable. Webhook events can be written without workspace attribution, making them impossible to debug or attribute to a tenant.

Finding 7-E (Medium): The leads table has no unique constraint on (workspace_id, phone). The sync deduplicates by meta->wbah_external_id then falls back to phone. If two WeeBespoke leads have the same phone but different external IDs, they both get stored — but only the first is found on the phone fallback path. Subsequent syncs may create duplicates.

Finding 7-F (Low): No index on leads.meta->>'wbah_external_id'. Every sync tick that does a lookup by external ID performs a full table scan on the JSONB meta column. At 1,568 rows this is fast; at 50,000+ it will be slow.

SECTION 9 — API ENGINE AUDIT
Finding 9-A (High): WeeBespoke API call history endpoint returns pageSize: 10 — hardcoded on the server side. To fetch 10,966 calls requires 1,097 API pages. The sync processes 20 pages per batch. Each batch makes 20 sequential HTTP requests. Full sync time ≈ 55 batches × (API latency × 20 requests) — likely 10–20 minutes per full sync cycle.

Finding 9-B (High): The leads sync fetches all 1,568 leads (32 pages at pageSize 50) but the calls sync fetches all 10,966 calls (1,097 pages at pageSize 10). Both run every 5 minutes. If the calls sync hasn't finished its previous cycle when the next one starts, two concurrent syncs run against the same wbah_calls table. There is no lock or mutex.

Finding 9-C (Medium): wbahGetLeadsFiltered (Disqualified endpoint) is capped at Math.min(totalPages, 100) pages — a maximum of 5,000 disqualified leads. If the actual count exceeds this, the People > Disqualified tab will silently show a truncated list.

Finding 9-D (Medium): Token refresh throttle (_wbahReloginAt, 30-minute cooldown) is a module-level variable. In the dev server, HMR reloads reset this variable. Each hard reload could trigger a re-login even within the 30-minute window.

SECTION 10 — PROVIDER HEALTH AUDIT
Finding 10-A: The health sweep runs every 15 minutes and writes results to provider_settings. This is the source of truth for the provider status indicators in the UI.

Finding 10-B (Medium): derivedConnected in the health check ORs dbConnectedSet — meaning if credentials are saved to DB, the provider shows as "connected" even if the health check itself fails. This can mask broken connections.

Finding 10-C (Low): WBAH does not use Retell, Twilio, or WhatsApp providers. The health sweep still checks them for the WBAH workspace if those providers are registered. These checks return failures that don't represent actual WBAH issues.
