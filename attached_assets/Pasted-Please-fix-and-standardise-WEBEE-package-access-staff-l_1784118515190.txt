Please fix and standardise WEBEE package access, staff login access, and Account Settings visibility controls.

The goal is to make new signups automatically receive access only to the areas of WEBEE included in their package from the website/pricing setup, while allowing workspace owners/admins to add staff users as a chargeable add-on and control what those staff members can see inside the app.

This must be workspace-scoped, secure, and must not affect WBAH or any other workspace incorrectly.

---

# 1. Main goal

Create a proper access control system based on:

1. Workspace package / subscription plan
2. Paid add-ons
3. Number of allowed staff users
4. User role
5. Page-level access
6. Action-level permissions
7. Owner/admin visibility settings

A new signup should not automatically access the full WEBEE system unless their package allows it.

---

* allowed pages

* allowed number of campaigns
* allowed number of staff users

# 3. Do not hardcode package rules in UI only

Do not only hide buttons on the frontend.

Package access must be enforced on the backend as well.

Every sensitive server action should check:

* workspace_id
* current package
* active subscription status
* allowed feature
* allowed action
* user role/permission
* add-on entitlement

If not allowed, return a clear blocked response:

“This feature is not included in your current package.”

or

“You need the Staff Access add-on to add more users.”

# 5. Staff logins as chargeable add-on

Staff logins should be chargeable.

Each package should include a base number of users.

Example:

* package includes 1 user
* extra staff users require add-on
* owner/admin can invite staff only up to included + paid add-on allowance

Add a Staff Access add-on.

Suggested addon:

`extra_staff_user`

Config:

* charged per extra user
* quantity based
* workspace-scoped
* visible in Account Settings / Billing
* enforced before invitation is accepted or activated

If a workspace tries to invite more users than allowed, show:

“You have reached your included staff user limit. Add extra staff users to continue.”

Do not silently allow unlimited staff logins.

# 10. Package gating for new signups

When a new workspace signs up:

1. Detect selected package.
2. Create workspace.
3. Create workspace subscription record.
4. Apply package entitlements.
5. Set included staff user limit.
6. Set page access based on package.
7. Set feature access based on package.
8. Set allowed AI departments based on package.
9. Hide/disable locked features.
10. Backend-block locked actions.
11. Show upgrade prompts for locked features.

If package is missing, default to lowest safe package or trial package.

Do not give full access by default.

---

# 11. Package definitions should match website pricing

Package definitions should be sourced from the existing pricing/package configuration used by the website or admin pricing system.

Do not hardcode old pricing values into multiple places.

Create a single source of truth for package rules.

If pricing currently only exists on the marketing website, create a synced internal package config and document how it is maintained.

The system should allow package rules to be updated without editing multiple files.

---

# 12. Locked feature UI

When a user tries to access a locked feature, show a clean upgrade state.

Example:

“This feature is not included in your current WEBEE package.”

Buttons:

* View Packages
* Request Upgrade
* Contact Admin
* Back to Dashboard

For staff users:

“You do not have permission to access this area. Ask your workspace admin for access.”

Do not show technical errors.

---

# 13. Billing/add-on handling

Staff logins must be chargeable as add-ons.

If billing integration exists, connect extra staff users to the billing provider.

If billing provider integration is not ready, create internal add-on records and mark billing integration as pending.

Do not block the whole system if billing provider is not connected, but do not silently make paid add-ons free unless explicitly in admin override/dev mode.

Add Account Settings → Billing display:

* current package
* included staff users
* active staff users
* extra staff users
* staff add-on cost
* active add-ons
* locked features
* upgrade options

# 14. Backend enforcement

Add a central permission/entitlement helper.

Suggested functions:

* `getWorkspacePackage(workspaceId)`
* `getWorkspaceEntitlements(workspaceId)`
* `canAccessFeature(workspaceId, userId, featureKey)`
* `canAccessPage(workspaceId, userId, pageKey)`
* `canPerformAction(workspaceId, userId, actionKey)`
* `canInviteStaff(workspaceId)`
* `getStaffSeatUsage(workspaceId)`
* `requireFeatureAccess(featureKey)`
* `requireActionAccess(actionKey)`
* `requirePageAccess(pageKey)`

Use these checks in server functions for:

* creating agents
* deploying agents
* purchasing numbers
* assigning SIP
* activating campaigns
* creating workflows
* using SystemMind
* creating reports
* creating People views
* creating page filters
* inviting staff
* changing settings
* accessing billing
* accessing provider settings

# 15. SystemMind must respect package and role access

SystemMind should not perform actions that the workspace package or user role does not allow.

Examples:

* If package does not include Campaigns, SystemMind cannot create a live campaign.
* If package does not include SystemMind, user cannot use SystemMind beyond allowed trial/preview.
* If user lacks Go Live permission, SystemMind can only create a draft and request approval.
* If staff add-on is not active, SystemMind cannot invite extra staff.
* If package does not include WhatsApp, SystemMind can suggest upgrade but not activate WhatsApp workflow.

SystemMind should explain:

“This package does not include this feature.”

or

“You do not have permission to apply this. I can create a draft for an admin to approve.”

---

# 18. Fail closed

If package or permission data cannot be loaded:

* do not grant access by default
* show a safe error
* allow owner/admin to access Account Settings only if safe
* log the error

Never fail open.

---

# 19. Do not break existing users

Migrate existing workspaces safely.

For existing active workspaces:

* assign current equivalent package or admin override
* preserve current access initially where required
* do not lock out owners
* do not break WBAH
* do not remove access without migration review
* create a migration report

For WBAH:

* preserve existing admin access
* add staff access controls carefully
* do not change current CRM/data access until roles are configured
