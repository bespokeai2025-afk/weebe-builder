# Analytics Hub (Analytics Centre BI + reporting upgrade)

- Two tables (migration `20260729000000_analytics_hub.sql`, applied via `scripts/apply-analytics-hub-migration.mjs`): `analytics_reports`, `analytics_report_schedules`. RLS: members read, service-role writes.
- Feature keys: `analytics_advanced`, `analytics_campaign_reports`, `analytics_financial`, `analytics_scheduled_reports`, `analytics_ai_insights`, `automated_report_emails` — gated server-side in `analytics-hub.functions.ts` / `reports.functions.ts`; UI tab locks are cosmetic only.
- Aggregation layer (`analytics-hub.server.ts`) is fail-closed: every helper catches and returns zeroed structures with an `error` field; leads table only counted head:true or window-bounded limit(1000); WBAH branches read `wbah_calls` and hide campaign/workflow/lead-source/follow-up sections.
- Report generator (15 types) NEVER throws — returns row id or null, so campaign lifecycle hooks are fire-and-forget. WBAH refuses campaign-lifecycle report kinds.
- **Schedule tick claim pattern**: `processAnalyticsReportSchedules` (campaign-executor tick) claims each due schedule with a compare-and-set update on `last_run_at` (`.eq` prior value or `.is null`) BEFORE generating/sending. **Why:** Supabase writes return `{error}` without throwing; treating a failed `last_run_at` write as success re-sends the report every 5-min tick (email spam). **How to apply:** any future scheduled-send sweep must claim-first with CAS, and check `{error}` on every state write.
- Email path: gate `automated_report_emails` → provider priority handled by `sendWorkspaceEmail` (workspace custom → reseller parent → WEBEE Resend); all user values escaped.
- e2e: `tests/e2e/analytics-hub.e2e.test.ts` (random-UUID workspace, cleans up).
- **v2 per-tab filter honesty rule**: shared FilterBar (agent/campaign/source) renders ONLY the selects a tab honors server-side (`TAB_FILTER_SUPPORTS` map in the analytics route). **Why:** filters that refetch but aren't applied in the aggregation helper produce silently wrong BI numbers (architect fail finding). **How to apply:** when adding a filter to a tab, first apply it inside the server helper, then flip its flag in the map. `calls` has no campaign_id column, so campaign filtering only works on the Campaigns tab.
- Booking anomaly detection: leads still `need_to_call` that already have a `calendar_bookings` row — looked up by booked lead ids only (never row-scan the giant leads table).
- Master Admin analytics rows carry `packageKey` + reseller parent (from `workspace_subscriptions` + active `workspace_relationships`); package/reseller filters are client-side over the returned rows.
- Nav is two-level: top sections Analytics | Sales | Marketing (`TAB_GROUPS` + `GROUP_OF` in the analytics route); Sales = leads/bookings/financial pills, everything else = pills under Analytics; `mainTab` stays the single source of truth, group is derived.
- WBAH Campaigns tab is a dialler report, not "not available": `getCampaignAnalyticsData` returns `mode:"wbah_dialler"` + `wbah` payload aggregated from `wbah_calls` (paged 1000/page, 25-page cap, `truncated` flag surfaced in UI). Voicemail = disconnection/end reason contains "voicemail". **Why:** WBAH dials via external WeeBespoke so WEBEE campaign KPIs don't exist; users still need converted/positive leads, voicemail and disconnection breakdowns. **How to apply:** any new WBAH-mode payload must stay workspace-scoped and keep the standard path untouched; the Campaigns tab entry in MAIN_TABS must NOT be `campaignStyle` (that hides it for WBAH).
- Automated campaign-report email setup: `ensureAutomatedCampaignReportSchedule` (report-schedule-setup.server.ts) is the ONE idempotent entry point — WBAH → `wbah_dialler_summary` report kind (dialler KPIs + converted leads; refused for non-WBAH workspaces), others → `daily_campaign_summary`; recipients default to owner+admin profile emails. SystemMind exposes it as `systemMindSetupCampaignReportSchedule` (gated analytics_scheduled_reports + notification_settings). **Why:** duplicating schedule-insert logic per caller caused divergent recipient/filter defaults. **How to apply:** any future "auto-email report" feature should call the ensure helper, not insert schedule rows directly; daily schedules should use dateFilter "yesterday" so each send covers a complete day.
- Campaigns tab includes a "Today's Schedule" section: `getCampaignAnalyticsData` parses each campaign's `__sched_v1__` description via the scheduler's `parseConfig` and returns `schedule` rows (callTime/timezone/frequency/lastRunDate/ranToday/runsToday). **Why:** users expect the analytics Campaigns view to show which daily campaigns run today, not only historical KPIs.
